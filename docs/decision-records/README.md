# Architecture Decision Records (ADRs)

Short, dated records of significant design decisions: the context, the options considered, the choice, and the consequences. ADRs make the *reasoning* durable, so a future reader (or you in six months) understands why — not just what.

## Template (`adr-NNN-title.md`)
```
# ADR-NNN: <decision>
Date: YYYY-MM-DD · Status: proposed | accepted | superseded
## Context
What forces are at play? What requirement/constraint triggered this?
## Options
The alternatives considered, with trade-offs.
## Decision
What we chose and why.
## Consequences
What this makes easy, hard, or risky going forward.
```
The [Service Decision Framework](../../SERVICE-DECISION-FRAMEWORK.md) is the menu; ADRs record which dish you picked for a given project and why.
